# NayzFreedom Fleet Bible v1

**Status:** Working canon  
**Purpose:** Define the shared world, hierarchy, vocabulary, and design principles for the NayzFreedom operating system before expanding the current dashboard into a multi-ship command center.

---

## 1. Core Idea

**NayzFreedom** is not merely a brand or a dashboard. It is a fleet led by **Captain Nayz** in pursuit of a freer life.

The fleet exists to support five core freedoms:

1. **Work** — freedom in how meaningful work is chosen and built
2. **Wealth** — financial independence and stewardship
3. **Love** — relationships, belonging, and emotional richness
4. **Health** — physical and mental vitality
5. **Soul** — art, meaning, spirituality, and inner truth

Together these are the **Freedom Five**, the deepest value system of the fleet.

## 3. The Captain

## Captain Nayz

**Role:** Owner, visionary, and final decision-maker of the fleet.

The system should help the Captain:

- see the whole picture quickly
- know what needs attention now
- make fewer but better decisions
- keep long-range goals visible
- avoid carrying every open loop mentally

The system should not force the Captain to:

- micromanage every task
- remember every deadline manually
- inspect every subsystem separately
- lose strategic direction inside daily noise

---

## 4. Fleet Command

## Nami

**Primary role:** Personal Secretary to the Captain  
**Scope:** Fleet-level visibility, captain support, and stewardship of `The Freedom`

Nami is not simply another crew member. She is the closest operational extension of the Captain.

### Responsibilities

- daily captain briefings
- personal priorities and reminders
- cross-ship visibility
- identifying decisions that require the Captain
- maintaining alignment with the Freedom Five
- overseeing personal domains within `The Freedom`
- surfacing conflicts between work, money, health, love, and soul

### Boundary with Robin

- **Nami** asks: “What matters most for the Captain’s life?”
- **Robin** asks: “What should The Aurora execute next?”

Nami sees the fleet. Robin runs the work ship.

## 7. Shared Vocabulary

The theme should enrich the system, not obscure it. Use thematic names for high-level places and ordinary language where clarity matters.

| Standard term | Fleet term |
|---|---|
| Dashboard home | Captain's Deck |
| Organization | Fleet |
| Business line / operating area | Ship |
| Brand project | Island |
| Personal domain | Domain |
| Job / workflow run | Mission or Voyage |
| Team | Crew |
| Alert | Distress Signal |
| Report | Captain's Brief |
| Goal | Horizon |
| Timeline / activity history | Logbook |
| Travel program | Horizon Atlas |

### Language principle

- Use **Captain's Deck** as a page title.
- Use **Mission** and **Voyage** where they create emotional resonance.
- Keep concrete labels like `Revenue`, `Budget`, `Release Date`, and `Portfolio Allocation` plain and obvious.

## 8. Cross-Ship Relationships

The fleet is not three unrelated products. Ships stay distinct but should coordinate.

### Examples

- `The Lyra` releases a new song → `The Aurora` may create a promotion campaign
- `The Freedom` flags low energy or a major life event → `The Aurora` may reduce mission load
- `The Aurora` generates new income → `The Freedom` may surface wealth decisions
- travel under `The Horizon Atlas` inspires songs in `The Lyra` or content in `The Aurora`

### Design requirement

The system should eventually support:

- linked missions
- cross-ship dependencies
- ship-to-ship references
- captain-level alerts
- one source of truth with multiple contextual views

## 9. Character and IP Guidance

The fleet may be inspired by the emotional energy of adventurous crew stories, but the world must remain original to NayzFreedom.

### Principles

- original character designs
- original costumes, symbols, and backstories
- no direct copying of existing anime characters, logos, or silhouettes
- if commercialized later, use full original identities such as `Nami Tide` or equivalent surnames if desired

## 11. Design Principles

1. **Clarity before theme**  
   Beautiful metaphors should never hide the answer the Captain needs.

2. **One source of truth**  
   Data may appear in multiple ships, but it should not be duplicated as separate conflicting records.

3. **Distinct ships, connected fleet**  
   Ships have their own workflows, but Captain-level oversight should unify them.

4. **The Captain sees decisions, not noise**  
   The interface should elevate next actions, blockers, and tradeoffs.

5. **Build for expansion without overbuilding now**  
   Preserve room for `The Freedom` and `The Lyra` while implementing the first usable version around `The Aurora`.

6. **History matters**  
   The system should preserve completed missions, decisions, releases, travels, and milestones as part of the Captain's logbook.

## 12. MVP Boundary

The first implementation should **not** attempt to fully build all ships at once.

### MVP should include

- fleet-aware navigation
- Captain's Deck shell
- upgraded `The Aurora` dashboard
- crew roster and character cards
- mission / voyage workflow view for existing content jobs
- scaffolding placeholders for `The Freedom` and `The Lyra`

### Defer for later

- full personal finance system
- full investment engine
- full song catalog manager
- release distribution workflows
- deep travel planner
- cross-ship automation
- advanced permissions
